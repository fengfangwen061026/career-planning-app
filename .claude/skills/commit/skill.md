# Commit Skill

当用户执行 `/commit` 命令时，自动完成以下操作：

1. **检查 git 状态**：运行 `git status` 获取所有修改的文件
2. **添加文件**：自动执行 `git add -A` 添加所有修改的文件
3. **生成 commit 消息**：
   - 运行 `git diff --cached` 查看暂存的更改
   - 根据修改内容生成简洁的 commit 消息（1-2 句话）
   - 遵循项目的 commit 消息风格
4. **执行提交**：使用生成的 commit 消息执行 `git commit`
5. **推送到远程**：执行 `git push` 推送到远程仓库
6. **上下文分析**：提交完成后，分析当前上下文内容，询问用户重点保留哪个分类

## 执行步骤

按顺序执行以下 bash 命令：

1. `git status` - 查看修改状态
2. `git diff --cached --stat` - 查看暂存文件的统计
3. `git add -A` - 添加所有修改
4. `git commit -m "commit message"` - 提交更改
5. `git push` - 推送到远程

如果当前分支没有远程追踪分支，先执行 `git push -u origin HEAD` 建立追踪关系。

## 上下文分析

提交推送成功后，**必须**分析当前上下文内容，然后询问用户。

### 第一步：分析上下文

扫描并识别当前上下文中包含的内容类型：

1. **打开的文件**：读取当前打开的编辑器标签页和文件路径
2. **最近的修改**：分析 git diff 或最近 commit 的内容
3. **对话主题**：识别正在讨论的技术话题、功能模块、问题类型
4. **记忆文件**：检查 `.claude/memory/` 目录中有哪些记忆文件
5. **项目状态**：检查项目进度、当前任务、待办事项

### 第二步：动态生成选项

根据分析结果，动态生成记忆保留选项。如果某个分类在上下文中没有相关内容，则不显示该选项。

#### 可能的分类

| 分类 | 内容来源 | 说明 |
|------|----------|------|
| 🎯 当前任务 | 当前打开的文件、最近的代码修改 | 进行中的功能、bug修复、遇到的问题 |
| 🗺️ 技术架构 | 架构相关的文件、模块划分 | 系统设计、技术选型、模块关系 |
| 📋 待办事项 | 记忆文件中的任务列表 | 功能计划、bug列表、优化项 |
| 📖 项目文档 | 文档文件、需求说明 | 需求分析、API文档、变更记录 |
| 💭 工作习惯 | 记忆文件中的偏好设置 | 代码风格、常用命令、快捷设置 |
| 🔧 环境配置 | IDE配置、开发环境文件 | 环境变量、调试配置、工具链 |
| 🐛 Bug追踪 | 错误日志、问题描述 | 已知问题、复现步骤、解决方案 |
| 💬 讨论上下文 | 对话历史中的关键讨论 | 技术决策、方案讨论、反馈意见 |

### 第三步：展示选项

基于分析结果，展示动态生成的选项：

```
✅ 推送成功！

分析当前上下文，发现以下内容：

📊 上下文内容概览：
├─ 📁 打开的文件：src/components/JobGraph/index.tsx, src/pages/StudentProfile.tsx
├─ 🐛 最近问题：图谱节点点击事件处理、样式兼容性
├─ 💬 技术讨论：关于 D3.js 图谱性能优化
├─ 📋 待办：完成岗位匹配功能、优化图谱渲染
└─ 📖 相关文档：graph.py 服务设计文档

请选择需要保留的记忆分类（点击选择）：

🎯 当前任务
   └─ 保留进行中的功能开发和问题细节

🗺️ 技术架构
   └─ 保留系统设计和模块关系

📋 待办事项
   └─ 保留功能计划和任务列表

📝 快捷操作
   ├─ [保留全部] - 不压缩，保留所有上下文
   └─ [稍后再说] - 暂不处理，下次 commit 时再问
```

### 展示规则

1. **只显示有内容的分类**：如果某个分类在上下文中不存在，不显示该选项
2. **优先显示最相关的**：根据当前上下文把最可能需要的排在前面
3. **提供快捷选项**："保留全部"和"稍后再说"
4. **使用可点击按钮**：让用户直接点击选择，不要用数字输入

## 压缩执行

1. 根据用户选择的分类，确定需要保留的内容范围
2. 保留选中分类的完整上下文
3. 删除未选中分类的详细内容，但保留关键摘要
4. 确保不丢失重要的工作记忆

## 注意事项

- 如果没有修改的文件，输出提示信息并跳过提交
- 如果远程推送失败（如需要 pull 先），提示用户处理
- commit 消息应该简洁明了，说明"做了什么"而不是"怎么做的"
- **必须**在推送成功后分析上下文并询问压缩选项
- 选项要动态生成，只显示当前上下文包含的分类
- 使用可点击按钮，不要用数字或文字输入
