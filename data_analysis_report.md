## 三、数据质量问题

### 3.1 重复数据
- **完全重复行**：527 条（占 5.3%）
- **岗位编码重复**：780 条（同一编码对应多行，说明数据采集时有重复抓取）

### 3.2 缺失值
| 字段 | 缺失数 | 缺失率 |
|------|--------|--------|
| 公司类型 | 2,975 | 29.9% |
| 公司详情 | 1,345 | 13.5% |
| 岗位详情 | 151 | 1.5% |
| 所属行业 | 2 | ~0% |
| 公司规模 | 2 | ~0% |

### 3.3 格式不一致
- **薪资范围**：三种单位混用
  - 月薪（元）：7,251 条，如 `4000-6000元`
  - 月薪（万）：2,222 条，如 `1-2万`
  - 日薪（元/天）：485 条，如 `120-150元/天`
  - 含 N 薪信息：1,657 条，如 `7000-10000元·13薪`
- **更新日期**：三种格式并存
  - `X月X日`（主体，无年份）
  - `YYYY-MM-DD HH:MM:SS`
  - `YYYY年MM月DD日`
- **地址**：336 条区县为 `None`（如 `郑州-None`）

### 3.4 内容质量
- **岗位详情**中 681 条（6.8%）内容不足 10 字符，仅为 `岗位职责:` `岗位职责：` `岗位要求` 等空壳
- **所属行业**存在同值重复拼接：`计算机软件,计算机软件`、`物联网,物联网`、`保险,保险`（共涉及数百条）
- **岗位详情**包含 HTML 标签 `<br>`，需清理

### 3.5 采样偏差
- 51 个岗位名称中，前 9 个技术类岗位（实施工程师/科研人员/技术支持/Java/测试/前端/C++/硬件测试）占 **4366 条（43.8%）**
- 其余 42 个岗位各约 147 条，疑似按固定配额采样
- 单一公司"外企德科"占 499 条（5%），为人力外包公司发布的岗位

## 五、技能关键词提取策略

### 5.1 数据来源
- 主要从 **岗位详情** 字段提取
- 辅助参考 **岗位名称** 和 **所属行业**

### 5.2 提取方法
1. **预处理**：去除 HTML 标签（`<br>`）、去除编号前缀（`1.` `2、`）、统一标点
2. **规则提取**：基于预定义的技能词典（编程语言、框架、工具、证书等）做正则匹配
3. **LLM 提取**：对规则未覆盖的岗位，使用 LLM 从职责描述中提取结构化技能列表
4. **技能归一化**：`JS` → `JavaScript`，`py` → `Python`，`PPT` → `Microsoft PowerPoint`

### 5.3 技能分类维度
- **硬技能**：编程语言、框架/库、数据库、工具/平台、证书/资质
- **软技能**：沟通能力、团队协作、项目管理
- **学历要求**：从岗位详情中提取最低学历
- **经验要求**：提取工作年限要求

## 六、清洗策略

### 6.1 字段取舍

| 字段 | 决策 | 理由 |
|------|------|------|
| 岗位名称 | **保留** | 核心字段，用于 Role 归一化 |
| 地址 | **保留，拆分** | 拆为 `city`（城市）+ `district`（区县），`None` 值置空 |
| 薪资范围 | **保留，解析** | 解析为 `salary_min`、`salary_max`（统一为月薪/元）、`salary_months`（年薪月数） |
| 公司名称 | **保留** | 用于去重和公司维度分析 |
| 所属行业 | **保留，清洗** | 去重拼接值（`计算机软件,计算机软件` → `计算机软件`），拆为行业列表 |
| 公司规模 | **保留** | 已是规范枚举值 |
| 公司类型 | **保留** | 缺失率 30% 可接受，填充为 `未知` |
| 岗位编码 | **保留** | 用作唯一标识和去重依据 |
| 岗位详情 | **保留，清洗** | 去除 HTML 标签，过滤空壳记录 |
| 更新日期 | **保留，标准化** | 统一为 `YYYY-MM-DD` 格式 |
| 公司详情 | **保留** | 可提取公司标签/特征，缺失可置空 |
| 岗位来源地址 | **丢弃** | 爬虫溯源用，与画像/匹配无关 |

### 6.2 清洗步骤

```
1. 去重：基于岗位编码去重（保留最新记录），预计去除 ~780 条
2. 过滤空壳：岗位详情 <10 字符的记录标记为低质量，酌情丢弃
3. 薪资解析：
   - "4000-6000元"        → min=4000, max=6000, unit=月
   - "1-2万"              → min=10000, max=20000, unit=月
   - "120-150元/天"       → min=2640, max=3300, unit=月 (×22天)
   - "7000-10000元·13薪"  → min=7000, max=10000, months=13
4. 地址拆分："郑州-None" → city=郑州, district=null
5. 行业去重："计算机软件,计算机软件" → "计算机软件"
6. 日期标准化：统一为 YYYY-MM-DD（缺少年份的默认 2025 年）
7. 岗位详情清洗：去除 <br> 标签，trim 空白
8. Role 归一化：添加 role 和 sub_role 字段
9. 技能提取：从岗位详情提取 skills 列表
```

### 6.3 建议的目标 Schema

清洗后的表结构（PostgreSQL）：

```sql
CREATE TABLE job_positions (
    id              SERIAL PRIMARY KEY,
    job_code        VARCHAR(64) UNIQUE NOT NULL,  -- 岗位编码（去重键）
    title           VARCHAR(128) NOT NULL,         -- 原始岗位名称
    role            VARCHAR(64) NOT NULL,          -- 归一化 Role 大类
    sub_role        VARCHAR(64),                   -- 细分方向（可选）
    city            VARCHAR(32) NOT NULL,          -- 城市
    district        VARCHAR(32),                   -- 区县
    salary_min      INTEGER,                       -- 最低月薪（元）
    salary_max      INTEGER,                       -- 最高月薪（元）
    salary_months   SMALLINT DEFAULT 12,           -- 年薪月数
    company_name    VARCHAR(256) NOT NULL,          -- 公司名称
    industries      TEXT[],                         -- 所属行业（数组）
    company_size    VARCHAR(32),                   -- 公司规模
    company_stage   VARCHAR(32),                   -- 融资阶段
    description     TEXT,                           -- 清洗后的岗位详情
    skills          TEXT[],                         -- 提取的技能关键词（数组）
    education_req   VARCHAR(32),                   -- 最低学历要求
    experience_req  VARCHAR(32),                   -- 工作经验要求
    company_intro   TEXT,                           -- 公司简介
    published_at    DATE,                           -- 发布/更新日期
    created_at      TIMESTAMP DEFAULT NOW()
);

-- 为匹配查询建索引
CREATE INDEX idx_job_role ON job_positions(role);
CREATE INDEX idx_job_city ON job_positions(city);
CREATE INDEX idx_job_skills ON job_positions USING GIN(skills);
CREATE INDEX idx_job_industries ON job_positions USING GIN(industries);
```

## 七、总结与建议

1. **数据整体质量中等偏上**：核心字段（岗位名称、薪资、地址、公司）完整度高，主要问题在于格式不统一和部分重复
2. **岗位名称已做初步归类**（51 类），可进一步归并为 ~25 个 Role 大类，覆盖大学生常见就业方向
3. **优先清洗薪资和日期字段**，这两个字段的格式混乱最影响后续匹配和分析
4. **技能提取建议分两轮**：先用规则词典快速覆盖 70%，再用 LLM 处理长尾
5. **建议补充数据**：当前数据偏向技术类岗位（占比近半），如需覆盖更多大学生就业方向，可补充教育、医疗、金融等行业数据
